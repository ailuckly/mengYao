# 文件存储设计

## 1. 设计目标

考虑到本课题属于本科毕业设计，系统规模较小，当前版本不强制引入数据库，而采用本地文件方式进行数据管理。文件存储设计的目标，是保证训练数据、配置文件、上传图片、识别结果和日志记录都有固定位置，便于开发、调试、测试和论文截图整理。

## 2. 存储结构设计

### 2.1 数据目录

`data/` 目录主要用于训练阶段的数据管理，内部划分如下：

- `data/raw/`：原始采集图片。
- `data/processed/`：清洗和整理后的图片。
- `data/annotations/`：标注文件。
- `data/splits/`：训练集和测试集划分结果。

### 2.2 模型目录

`model/` 目录主要用于训练配置和权重文件管理：

- `model/configs/`：类别配置、建议映射、训练配置。
- `model/checkpoints/`：训练后生成的模型权重。
- `model/scripts/`：训练和推理脚本。
- `model/exports/`：导出模型文件。

### 2.3 运行记录目录

`records/` 目录主要用于系统运行阶段的数据：

- `records/uploads/`：用户上传的原始图片。
- `records/predictions/`：推理后生成的结果图。
- `records/logs/`：系统日志文件。

## 3. 文件命名规范

### 3.1 上传文件命名

为了避免重名覆盖，用户上传文件采用“随机 ID + 原始文件名”的方式保存。

示例：

`e1f3a2b4c5_sample.jpg`

### 3.2 结果图命名

识别结果图采用“固定前缀 + 随机 ID”的方式生成。

示例：

- `predict_xxxxx.png`
- `mock_xxxxx.png`

### 3.3 配置文件命名

配置文件采用语义化命名，当前主要包括：

- `label_advice.json`

## 4. 当前不使用数据库的原因

当前系统主要完成图片识别和结果展示，不涉及用户登录、历史查询、复杂统计报表等功能。因此，使用数据库会增加实现复杂度，不利于控制开发周期。采用本地文件方式已经能够满足：

- 图片保存
- 结果图存档
- 配置读取
- 演示与测试

如果后续需要扩展用户记录管理，可以再引入 SQLite 或 MySQL。

## 6. 本章小结

本章完成了系统的文件存储设计，明确了训练阶段、运行阶段和配置阶段的目录职责。该设计结构简单、易维护，能够满足当前毕业设计项目的实现需求。
