# 测试方案设计

## 1. 测试目标

系统测试的目标是验证家庭园艺植物识别系统在功能、性能和稳定性方面是否满足项目预期，并为毕业设计说明书中的测试章节提供可直接使用的测试依据。

## 2. 测试范围

本系统当前测试范围包括：

- 页面上传功能
- 图片预览功能
- 后端接口可用性
- 推理结果返回
- 结果页面渲染
- 异常情况处理
- 响应时间测试

## 3. 测试环境

建议测试环境如下：

- 操作系统：Windows 或 Linux
- 浏览器：Chrome、Edge
- 后端环境：Python + Flask
- 模型环境：YOLOv11 或占位推理模式

## 4. 功能测试设计

### 4.1 功能测试项

| 编号 | 测试内容 | 预期结果 |
|---|---|---|
| F1 | 上传 JPG 图片 | 上传成功并进入识别流程 |
| F2 | 上传 PNG 图片 | 上传成功并进入识别流程 |
| F3 | 上传非法格式文件 | 系统提示格式不支持 |
| F4 | 上传空文件 | 系统提示文件无效 |
| F5 | 正常识别植物图片 | 返回类别、置信度和建议 |
| F6 | 正常识别病害图片 | 返回病害类别、置信度和建议 |
| F7 | 多次连续上传图片 | 系统保持稳定运行 |
| F8 | 调用 `/health` 接口 | 返回服务正常状态 |
| F9 | 调用 `/labels` 接口 | 返回系统支持类别列表 |

## 5. 性能测试设计

### 5.1 性能指标

系统性能主要从以下几个方面评估：

- 单张图片识别耗时
- 接口平均响应时间
- 页面展示耗时

### 5.2 性能测试方法

- 选取典型测试图片，重复调用识别接口。
- 记录每次请求开始和结束时间。
- 统计平均响应时间。
- 比较占位模式与真实模型模式下的差异。

## 6. 稳定性测试设计

稳定性测试主要关注以下情况：

- 连续上传多张图片时系统是否报错。
- 异常图片输入时系统是否中断。
- 后端在连续请求下是否保持可用。

## 7. 典型测试样例设计

建议测试图片包括：

- 清晰植物图片
- 光照不足图片
- 背景复杂图片
- 多目标图片
- 病害叶片图片
- 非植物图片

## 8. 测试结果记录格式

后续正式测试时，建议采用表格记录以下字段：

- 测试编号
- 测试时间
- 输入图片
- 识别结果
- 是否通过
- 备注

## 9. 本章小结

本章完成了系统测试方案设计，明确了测试目标、测试范围、功能测试和性能测试思路。后续在系统功能稳定后，可以直接按本章表格和流程补充真实测试结果。
