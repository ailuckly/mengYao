# 系统总体设计

## 1. 引言

在完成系统需求分析之后，需要进一步从整体角度对系统进行结构设计。系统总体设计的主要任务是明确系统架构、功能模块、数据流向、接口关系以及运行流程，为后续编码实现和测试验证提供统一依据。

本系统以家庭园艺植物识别为应用场景，采用“前端页面 + Flask 后端服务 + YOLOv11 模型推理”的总体结构。系统既要满足植物种类识别和病虫害检测需求，又要保证页面交互简洁、识别流程清晰、部署方式简单，因此总体设计上应尽量采用模块化和轻量化方案。

---

## 2. 系统设计目标

系统总体设计主要围绕以下几个目标展开：

1. 保证系统结构清晰，便于实现与维护。
2. 保证前后端分工明确，降低模块耦合度。
3. 保证模型推理流程稳定，便于后续优化和替换。
4. 保证页面交互简单直观，适合普通用户使用。
5. 保证系统能够满足毕业设计演示和论文撰写需要。

---

## 3. 系统总体架构设计

### 3.1 架构说明

本系统采用三层结构设计，分别为表示层、业务层和模型层。

1. 表示层  
   主要由 H5 页面组成，负责图片上传、结果展示和用户交互。

2. 业务层  
   主要由 Flask 后端服务组成，负责接收请求、调用模型、处理识别结果、组织返回数据。

3. 模型层  
   主要由训练完成的 YOLOv11 检测模型组成，负责对上传图片进行目标检测和分类判断。

这种结构的优点是职责划分较清晰：前端只负责交互，后端负责逻辑和接口，模型模块负责识别任务，便于后期调试和扩展。

### 3.2 系统总体架构图说明

系统整体工作流程如下：

1. 用户在前端页面上传图片。
2. 前端将图片发送到 Flask 后端。
3. 后端对图片进行校验和预处理。
4. 后端调用 YOLOv11 模型进行推理。
5. 模型返回检测框、类别名称和置信度。
6. 后端对结果进行整理，并匹配养护建议。
7. 前端页面展示识别结果、建议和标注图像。

可以概括为：

`用户 -> 前端页面 -> Flask接口 -> YOLOv11模型 -> 结果处理 -> 前端展示`

## 4. 系统功能模块设计

根据系统需求分析，本系统总体上可划分为六个核心模块。

### 4.1 图片上传模块

该模块负责接收用户上传的图片，并在前端进行基本预览。上传后由后端对文件进行进一步校验，包括文件类型、文件完整性和大小限制等。该模块是整个识别流程的入口。

主要功能包括：

- 选择图片
- 图片预览
- 文件上传
- 上传状态提示

### 4.2 图片预处理模块

后端在接收到图片之后，需要进行一定的预处理，以便模型能够稳定推理。预处理内容包括文件读取、格式转换、尺寸调整和异常数据过滤等。

主要功能包括：

- 图片合法性检查
- 格式统一
- 尺寸适配
- 无效图片拦截

### 4.3 模型推理模块

模型推理模块是系统的核心部分，主要负责加载训练完成的 YOLOv11 模型，并对输入图像进行检测。该模块输出图片中的目标框、类别名称和置信度。

主要功能包括：

- 模型加载
- 图片输入处理
- 目标检测
- 结果输出

### 4.4 结果处理模块

模型输出的原始结果通常是检测框、类别编号和置信度等信息，不能直接呈现给用户。因此需要结果处理模块对推理结果进行整理，包括类别映射、置信度转换、检测框绘制和建议匹配。

主要功能包括：

- 类别编号转文字标签
- 检测结果可视化
- 置信度整理
- 养护建议映射

### 4.5 结果展示模块

该模块由前端页面实现，负责将识别结果以用户容易理解的方式展示出来。展示内容应突出核心信息，不宜堆积过多技术细节。

主要展示内容包括：

- 原始图片或标注后图片
- 识别出的植物或病害类别
- 置信度
- 基础养护建议
- 错误提示或识别失败提示

### 4.6 日志与记录模块

为了方便调试、测试和答辩展示，系统应保留基本日志和部分识别记录。该模块不需要设计得非常复杂，但应满足演示和论文记录需求。

主要功能包括：

- 上传文件保存
- 识别结果记录
- 运行日志记录
- 错误日志记录

## 5. 系统流程设计

### 5.1 用户识别流程

用户识别流程如下：

1. 用户打开系统首页。
2. 用户进入识别页面并上传图片。
3. 系统对上传文件进行校验。
4. 合法图片进入推理流程。
5. 后端调用模型进行识别。
6. 系统整理结果并生成返回数据。
7. 前端展示识别结果和养护建议。

### 5.2 异常处理流程

当系统遇到异常情况时，流程如下：

1. 若图片格式错误，则前端或后端提示“文件格式不支持”。
2. 若图片为空或损坏，则提示“图片无效，请重新上传”。
3. 若模型推理失败，则提示“识别失败，请稍后重试”。
4. 若未检测到明显目标，则提示“请上传更清晰的植物图片”。

### 5.3 系统内部处理流程

从后端处理逻辑来看，内部流程如下：

1. 接收请求。
2. 保存上传图片。
3. 加载或调用已加载模型。
4. 执行推理。
5. 解析预测结果。
6. 匹配文字标签与养护建议。
7. 返回 JSON 数据或渲染页面结果。

## 6. 数据设计

### 6.1 数据组织方式

本系统的数据主要分为训练数据、运行数据和结果数据三类。

1. 训练数据  
   包括原始图片、标注文件、训练集和测试集划分结果。

2. 运行数据  
   包括用户上传图片、模型加载配置、类别映射表和建议映射表。

3. 结果数据  
   包括识别结果图、类别名称、置信度和日志信息。

### 6.2 类别映射设计

为了保证模型输出与页面展示一致，系统中需要建立类别映射表。该映射表用于将模型输出的类别编号转换为可读文字。

例如：

- `0 -> 绿萝`
- `1 -> 吊兰`
- `2 -> 多肉`
- `3 -> 叶斑病`
- `4 -> 黄化症状`

实际类别应根据训练数据集确定。

### 6.3 养护建议设计

系统应建立简单的建议映射表，根据不同识别类别返回不同提示信息。例如：

- 若识别为绿萝，则提示“注意散射光环境，避免长期暴晒，保持盆土湿润但不过湿”。
- 若识别为叶斑病，则提示“建议减少叶面潮湿时间，观察病斑扩散情况，必要时隔离病株”。

建议内容不需要设计得过于复杂，但应保持语言简洁、实用。

### 6.4 是否使用数据库

从本课题规模来看，系统前期可以不强制引入复杂数据库，而采用文件存储方式保存图片、识别结果和配置数据。这样实现更简单，也更适合毕业设计开发周期。如果后续需要扩展识别记录查询、用户管理等功能，再考虑引入 SQLite 或 MySQL。

## 7. 接口设计

### 7.1 接口设计原则

系统接口应尽量简单明确，便于前后端联调和后续调试。接口命名应清晰，返回数据结构应统一。

### 7.2 主要接口设计

#### 7.2.1 图片识别接口

- 接口路径：`/predict`
- 请求方式：`POST`
- 功能说明：接收前端上传图片，返回识别结果

请求参数：

- `file`：上传图片文件

返回参数建议包括：

- `status`：请求状态
- `message`：提示信息
- `labels`：识别类别列表
- `scores`：置信度列表
- `boxes`：检测框坐标
- `advice`：养护建议
- `result_image`：结果图片路径

#### 7.2.2 健康检查接口

- 接口路径：`/health`
- 请求方式：`GET`
- 功能说明：检查系统服务是否正常运行

#### 7.2.3 类别查询接口

- 接口路径：`/labels`
- 请求方式：`GET`
- 功能说明：返回当前支持识别的类别信息

## 8. 部署设计

### 8.1 本地部署方案

考虑到毕业设计的开发和演示需求，系统首先采用本地部署方式。前端页面和 Flask 服务部署在同一台设备上，模型文件保存在本地。该方案结构简单，便于调试和演示。

### 8.2 部署流程说明

部署时主要包括以下步骤：

1. 安装 Python 环境及依赖库。
2. 配置 PyTorch 和模型运行环境。
3. 放置训练完成的 YOLOv11 权重文件。
4. 启动 Flask 服务。
5. 通过浏览器访问系统页面。

### 8.3 部署要求

- 模型文件路径清晰。
- 图片上传目录可正常读写。
- 前后端访问路径配置正确。
- 演示环境中浏览器可正常访问。

---

## 9. 系统安全与异常设计

虽然本系统主要面向本地演示，不涉及复杂的用户权限控制，但仍需考虑基础安全性和异常处理问题。

### 9.1 文件安全

- 限制上传文件类型。
- 限制上传文件大小。
- 对上传文件名进行规范化处理。

### 9.2 服务安全

- 后端对非法请求进行拦截。
- 对接口异常统一返回错误信息。
- 对路径访问进行基本限制，避免无关文件泄露。

### 9.3 异常提示设计

系统应避免把底层报错直接显示给用户，而应转换为简洁友好的提示语。例如：

- “上传失败，请重新选择图片”
- “未检测到明显目标，请更换图片”
- “系统繁忙，请稍后再试”

---

## 10. 系统总体设计小结

本章从整体角度完成了家庭园艺植物识别系统的结构设计，明确了系统采用“前端页面 + Flask 后端 + YOLOv11 模型”的总体方案，并对功能模块、处理流程、数据组织、接口设计和部署方式进行了说明。

通过总体设计可以看出，本系统结构相对清晰，模块职责较明确，能够满足毕业设计中“模型识别 + 系统实现 + 页面展示”的基本要求。后续实现阶段将以本章设计内容为依据，进一步完成详细设计、功能开发、模型联调和测试分析。
