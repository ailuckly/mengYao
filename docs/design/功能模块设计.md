# 功能模块设计

## 1. 模块划分原则

本系统按“页面交互、接口服务、模型推理、结果处理、文件管理”五类职责进行模块划分。这样划分的目的，是减少模块之间的耦合，使系统在后续接入真实模型、增加类别或者调整页面样式时，不需要大面积改动。

## 2. 前端交互模块

### 2.1 模块功能

- 提供图片上传入口。
- 显示本地图片预览。
- 触发识别请求。
- 展示系统返回结果。

### 2.2 输入与输出

- 输入：用户上传的图片文件。
- 输出：识别请求和可视化结果页面。

## 3. 图片上传与校验模块

### 3.1 模块功能

- 接收用户上传的文件。
- 校验文件名和后缀。
- 拒绝非法类型文件。
- 将合法图片保存到指定目录。

### 3.2 输入与输出

- 输入：前端上传文件。
- 输出：本地图片路径。

## 4. 模型推理模块

### 4.1 模块功能

- 加载 YOLOv11 模型。
- 对输入图片进行推理。
- 输出识别框、类别和置信度。

### 4.2 输入与输出

- 输入：本地图片路径。
- 输出：预测结果列表。

## 5. 结果处理模块

### 5.1 模块功能

- 将类别编号转换为类别名称。
- 为识别结果生成建议文本。
- 生成结果图路径。
- 输出前端可直接使用的 JSON 数据。

### 5.2 输入与输出

- 输入：模型预测结果。
- 输出：标准化响应数据。

## 6. 日志与记录模块

### 6.1 模块功能

- 保存上传图片。
- 保存结果图。
- 记录系统运行日志。

### 6.2 输入与输出

- 输入：图片路径、预测结果、运行状态。
- 输出：本地记录文件。

## 7. 模块调用关系

系统模块调用顺序如下：

`前端交互模块 -> 图片上传与校验模块 -> 模型推理模块 -> 结果处理模块 -> 前端展示`

在这个流程中，日志与记录模块横向参与上传保存和结果保存两个环节。

## 8. 本章小结

通过对功能模块进行拆分，系统已经形成了清晰的实现边界。后续编码时，各模块可以独立实现、独立调试，再通过统一接口进行集成。
