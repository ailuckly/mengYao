# 系统需求分析

## 1. 引言

### 1.1 课题背景

本课题的题目为“基于YOLOv11的家庭园艺植物识别系统设计与实现”。随着家庭园艺逐渐普及，越来越多的用户开始在家中种植花卉、绿植和小型蔬果。相比专业种植人员，普通家庭用户通常缺乏植物分类和病虫害识别经验，遇到植物叶片发黄、长斑、枯萎或生长异常等情况时，很难快速判断原因，也不清楚应该采取什么处理措施。

在这种情况下，如果系统能够通过图像识别技术自动判断植物种类和常见病虫害，并向用户返回基础养护建议，就可以提高家庭园艺养护效率，也能减少因误判带来的时间和成本损失。因此，设计一个面向家庭园艺场景、具备实际可用性的植物识别系统，具有一定的应用价值和研究意义。

### 1.2 系统建设目标

本系统的建设目标是基于 YOLOv11 实现一个家庭园艺植物识别系统，使用户能够通过上传图片完成植物种类识别和常见病虫害检测，并在页面中查看识别结果、置信度以及基础养护建议。系统不仅要完成模型识别，还要实现前后端交互、结果展示和基础数据管理，形成完整的可运行应用。

### 1.3 需求分析的目的

需求分析是系统设计和实现的基础。通过对系统使用场景、目标用户、功能需求和性能需求进行梳理，可以明确系统的实现边界，避免在开发过程中出现功能重复、目标不清或者设计偏差的问题。同时，需求分析内容也将为后续系统总体设计、详细设计、功能实现和测试分析提供依据。

## 2. 可行性分析

### 2.1 技术可行性

本系统采用 Python 作为主要开发语言，使用 PyTorch 框架完成深度学习模型训练，使用 YOLOv11 作为核心目标检测模型，技术路线较为清晰。PyTorch 具有开发灵活、社区资源丰富、训练与调试方便等特点，适合完成毕业设计中的模型训练和实验比较。Flask 作为轻量级 Web 框架，结构简单，便于快速完成接口开发和页面整合，适合本课题的项目规模。

YOLO 系列模型在目标检测任务中已经得到了广泛应用，具有检测速度快、部署方便、适合实时识别场景等优点。结合本课题中家庭园艺植物识别的需求，采用 YOLOv11 作为核心模型具有较好的可行性。前端采用 H5 页面，可以兼顾电脑端和移动端访问，便于后续系统演示与答辩展示。

### 2.2 经济可行性

本系统开发主要依赖开源框架和现有计算机设备，不需要采购额外的商业软件授权。模型训练可基于现有实验环境完成，数据来源以公开数据、自采图片和网络补充数据为主，整体成本较低，符合本科毕业设计的条件。

### 2.3 操作可行性

系统面向的使用对象主要是普通家庭园艺用户，因此操作流程必须尽量简单。用户只需要完成图片上传，即可得到识别结果和基础建议，使用门槛较低。系统在设计时采用网页方式访问，也避免了复杂安装过程，具有较好的操作可行性。

## 3. 业务需求分析

### 3.1 目标用户分析

本系统的目标用户主要包括以下两类：

1. 家庭园艺普通用户  
   这类用户缺乏专业植物知识，希望通过简单操作快速识别植物种类和植物异常情况。

2. 毕业设计演示与教学使用者  
   在答辩和展示场景中，系统还需要面向教师或同学展示识别流程，因此系统应具备页面清晰、操作流畅、结果直观的特点。

### 3.2 业务场景分析

在实际使用中，用户打开系统页面后，上传一张植物图片。系统接收到图片后，调用后端识别服务，并将图片输入训练好的 YOLOv11 模型中进行推理。模型输出目标框、类别名称和置信度，系统再结合预设的养护建议库，对识别结果进行整理，最后在页面中展示给用户。

如果识别的是植物类别，系统可以返回植物名称及简要养护建议；如果识别的是病害类别，系统则可以返回病害名称、可能问题描述和基础处理建议。若图片模糊、目标不明显或系统无法判断，则应返回相应提示，提醒用户重新上传更清晰的图片。

### 3.3 业务流程分析

系统的基础业务流程如下：

1. 用户进入系统首页。
2. 用户选择或拍摄植物图片。
3. 前端将图片上传到后端服务。
4. 后端完成图片校验和模型推理。
5. 系统整理识别结果并匹配建议信息。
6. 页面展示植物类别、病害类别、置信度和养护建议。
7. 用户根据结果进行查看或重新上传图片。

## 4. 功能需求分析

### 4.1 总体功能需求

根据课题目标，系统需要完成以下几个方面的核心功能：

1. 图片上传功能。
2. 植物种类识别功能。
3. 病虫害检测功能。
4. 识别结果展示功能。
5. 养护建议输出功能。
6. 异常处理与错误提示功能。

### 4.2 具体功能需求

#### 4.2.1 图片上传功能

系统应支持用户从本地选择图片上传，后续如有条件也可扩展为直接拍照上传。上传前需要对文件类型、文件大小和图片有效性进行基本校验，避免非图片文件或损坏文件进入识别流程。

该功能的具体要求如下：

- 支持常见图片格式，如 JPG、JPEG、PNG。
- 支持图片预览。
- 上传失败时给出明确提示。
- 对空文件、损坏文件和非法格式文件进行拦截。

#### 4.2.2 植物种类识别功能

系统应能够识别训练集中包含的常见家庭园艺植物类别。当图片中存在目标植物时，系统应输出对应类别名称和置信度。如果一张图片中存在多个目标，也应尽量支持多目标识别和结果标注。

该功能的具体要求如下：

- 能识别预设植物类别。
- 能返回目标框和类别名称。
- 能给出识别置信度。
- 对同图多目标情况能够进行区分。

#### 4.2.3 病虫害检测功能

系统应支持对训练集中包含的常见病虫害症状进行检测，例如叶片病斑、卷叶、局部发黄等典型表现。病害识别应建立在样本数据可支撑的前提下，优先保证常见病害类别的检测效果。

该功能的具体要求如下：

- 支持识别系统定义的常见病害类别。
- 可对病害区域进行框选。
- 输出病害名称和置信度。
- 对识别不确定情况给出提示信息。

#### 4.2.4 识别结果展示功能

识别完成后，系统应在前端页面中直观展示结果，方便用户理解。展示信息应尽量简洁，不宜堆积过多技术细节。

展示内容包括：

- 上传图片。
- 检测标注结果图。
- 植物或病害类别名称。
- 置信度信息。
- 简单文字说明。

#### 4.2.5 养护建议输出功能

为了增强系统实用性，系统应在识别结果基础上提供简要养护建议。例如，当识别出某类植物时，可以提示浇水、光照、通风等基础建议；当识别出病害时，可以提示用户注意隔离、减少浇水或进一步观察。

该功能的具体要求如下：

- 根据类别返回对应建议。
- 建议内容简短明了。
- 建议与识别结果保持对应关系。

#### 4.2.6 异常处理功能

在实际使用中，系统可能会遇到上传失败、模型推理失败、图片无目标、图片模糊等情况，因此需要具备基本的异常处理能力。

该功能的具体要求如下：

- 上传异常时提示用户重新上传。
- 模型推理异常时返回统一错误信息。
- 无法识别时提示“请更换更清晰图片”或类似说明。
- 页面请求超时或服务异常时给出友好提示。

## 5. 非功能需求分析

### 5.1 性能需求

系统的性能需求主要体现在识别准确率和响应速度两个方面。根据任务书要求，目标检测精度应达到 90% 左右，同时单帧推理时间应尽量控制在 200ms 以内。考虑到毕业设计的实际开发条件，在系统实现过程中，应以“能稳定运行并具备较好实时性”为基本目标，再逐步优化模型表现。

性能方面主要要求如下：

- 模型在测试集上具有较好的识别准确率。
- 单张图片识别响应时间尽量短。
- 页面上传和结果返回流程流畅。
- 系统在连续多次识别时保持稳定。

### 5.2 易用性需求

系统使用对象主要是普通用户，因此页面设计应简洁清晰，操作步骤不宜过多。用户不需要了解复杂的模型原理，只需要知道如何上传图片和查看结果即可。

易用性方面主要要求如下：

- 首页功能明确。
- 上传入口清晰。
- 结果页面重点信息突出。
- 说明文字尽量通俗。

### 5.3 可靠性需求

系统需要在一般使用条件下保持基本稳定，避免频繁报错或页面卡死。对于无法识别的情况，系统也应有合理反馈，而不是直接中断流程。

### 5.4 可维护性需求

由于本系统后续还需要用于论文整理和答辩演示，因此在实现过程中应注意结构清晰、模块分离。后端接口、模型推理和前端页面应尽量独立，方便后续调试、修改和补充说明书内容。

### 5.5 可扩展性需求

虽然当前系统以本科毕业设计规模为主，但在设计时仍应考虑一定扩展性。例如，后续可以增加更多植物类别、增加用户识别记录、增加历史结果查看功能，或者将识别方式扩展到移动端拍照识别。因此，在需求设计时应尽量避免过度耦合。

## 6. 用例分析

### 6.1 主要参与者

- 用户
- 系统管理员（毕业设计实现中可不设置独立后台，仅作为后续扩展角色）

### 6.2 核心用例

#### 用例一：上传图片并识别植物

- 参与者：用户
- 前置条件：系统正常运行，用户已进入识别页面
- 基本流程：
  1. 用户选择图片。
  2. 系统预览图片并上传。
  3. 后端调用模型完成识别。
  4. 页面展示识别结果和建议。
- 后置条件：用户获得识别结果

#### 用例二：识别病虫害并查看建议

- 参与者：用户
- 前置条件：上传图片中存在病害目标
- 基本流程：
  1. 用户上传植物异常图片。
  2. 系统识别病害区域。
  3. 页面返回病害名称和处理建议。
- 后置条件：用户获得病害识别结果

#### 用例三：异常图片处理

- 参与者：用户
- 前置条件：上传图片格式错误、图片模糊或无有效目标
- 基本流程：
  1. 用户上传图片。
  2. 系统进行校验或推理。
  3. 系统返回错误提示或重新上传提示。
- 后置条件：用户了解失败原因并可继续操作

## 7. 数据需求分析

### 7.1 数据来源需求

本系统训练数据主要来源于以下三个部分：

1. 公开植物图像数据集。
2. 网络补充图像。
3. 自采家庭园艺场景图片。

### 7.2 数据内容需求

数据应覆盖以下几类信息：

- 常见家庭园艺植物类别。
- 常见病害类别。
- 不同拍摄角度。
- 不同光照条件。
- 不同背景环境。

### 7.3 数据质量需求

为了保证模型训练效果，数据应满足以下要求：

- 图像清晰度基本可用。
- 标注框位置准确。
- 类别命名统一。
- 不出现大量重复样本。
- 训练集与测试集划分合理。

本章围绕“基于YOLOv11的家庭园艺植物识别系统”进行了需求分析，主要从可行性、业务需求、功能需求、非功能需求、用例需求和数据需求几个方面对系统进行了梳理。通过分析可以看出，本系统的核心任务不仅是完成植物和病虫害识别，还需要保证识别流程简单、页面展示清晰、结果具有一定实用性。

需求分析的结果为后续系统总体设计提供了明确依据。下一步应在此基础上继续完成系统架构设计、模块划分、数据流程设计和功能实现方案设计，使系统从需求层面逐步过渡到工程实现层面。
