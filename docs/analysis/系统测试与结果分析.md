# 系统测试与结果分析

## 1. 测试目的

系统测试的目的在于验证家庭园艺植物识别系统在功能完整性、接口可用性、运行稳定性和结果可解释性方面是否满足毕业设计的实现要求，并为论文中的“系统测试与结果分析”章节提供真实依据。测试内容不仅包括前后端主流程，还包括历史记录、知识库、模型分析和异常处理等辅助模块。

## 2. 测试环境

本次测试采用本地部署方式完成，主要环境如下：

- 操作系统：Linux
- 开发语言：Python 3
- Web 框架：Flask
- 深度学习框架：PyTorch
- 识别模型：YOLOv11s
- 前端形式：Flask 模板 + 原生 JavaScript
- 本地数据库：SQLite
- 测试方式：自动化接口测试与人工页面联调相结合

系统默认加载 `model/checkpoints/best.pt` 作为正式演示模型，若权重不存在则自动退回占位推理模式，以保证页面主流程可正常运行。

## 3. 测试内容与方法

结合系统实现范围，本次测试主要分为以下三类：

### 3.1 功能测试

功能测试重点验证以下内容：

- 首页、识别页、历史记录页、知识库页、模型分析页是否可正常访问
- 图片上传与识别流程是否能够正常执行
- 识别结果图、类别、置信度与建议是否能够正确展示
- 历史记录是否能够在识别后自动写入并支持查看、删除
- 知识库与模型分析页是否能够正确读取配置与图表资源
- 非法文件上传、缺失记录访问等异常情况是否能正确处理

### 3.2 接口测试

接口测试主要围绕以下接口展开：

- `GET /health`
- `GET /labels`
- `POST /predict`
- `GET /api/history`
- `GET /api/history/<id>`
- `GET /api/knowledge`
- `GET /api/analysis`

接口测试采用项目内自动化测试脚本执行，并通过返回状态码、字段结构和关键数据内容判断接口是否正常。

### 3.3 性能与稳定性测试

性能与稳定性测试主要从以下角度进行验证：

- 单张图片识别时间是否在可接受范围内
- 多次调用识别接口时系统是否能够稳定返回结果
- 系统在加载真实模型时能否保持主流程可用
- 页面在无记录、无结果、异常请求等边界情况下是否能正常返回友好提示

## 6. 性能与稳定性分析

### 6.1 单张图片识别耗时

在本地联调和测试过程中，系统使用真实模型进行推理时，单张测试图片的模型推理日志显示耗时大约在 `100 ms` 到 `110 ms` 之间，预处理与后处理时间较短，能够满足本项目“单图识别、结果可视化展示”的应用需求。

需要说明的是，该时间与具体硬件环境、输入图片尺寸和运行方式有关，因此本文更关注“能够稳定完成识别并在合理时间内返回结果”这一目标，而不是追求严格的实时检测指标。

### 6.2 稳定性表现

从自动化测试和多次手动联调结果来看，系统在以下方面表现稳定：

- 页面切换和导航访问正常
- 单图上传与识别流程能够重复执行
- 识别后历史记录可正常落库
- 删除记录不会破坏系统其他功能
- 异常请求能够返回可解释信息，而不会导致系统崩溃

因此，当前系统已经具备本地展示、课程演示和毕业答辩所需的稳定性。

## 7. 识别结果综合分析

结合模型训练结果与系统联调表现，可以得到以下结论：

### 7.1 植物类别识别效果较好

系统采用的最终模型为 `YOLOv11s`，在训练批次 `colab_pro_formal_v13` 中取得了较稳定的结果。实验结果表明，6 类植物的识别效果整体较好，类别区分度较高，在系统页面中能够较稳定地返回主判定类别和结果图，适合作为毕业设计演示的核心功能。

### 7.2 病害与异常类别具有辅助识别能力

病害与异常类整体效果弱于植物类，主要原因在于样本数量相对较少、类别间差异不够明显、部分类别存在一定混淆。因此，系统更适合定位为“病害与异常辅助识别”，而不是专业诊断系统。这一定位与课题目标和实现条件基本一致。

### 7.3 当前结果能够支撑系统演示和论文分析

当前模型的关键指标如下：

- Precision：`0.7441`
- Recall：`0.7495`
- mAP50：`0.7279`
- mAP50-95：`0.6754`

从实验角度看，上述结果已经能够支撑论文中的模型训练、实验分析和系统展示章节；从应用角度看，系统已经能够完成“上传图片 -> 返回结果 -> 保存记录 -> 查看详情”的完整闭环。

## 8. 存在的问题与改进方向

虽然系统已经达到毕业设计实现目标，但从测试和实验结果来看，仍存在以下问题：

1. 病害与异常类的识别效果弱于植物类，部分类别仍有混淆。
2. 当前系统以单图识别为主，对复杂背景和多目标场景的适应能力仍有限。
3. 测试更多集中于核心流程，后续若继续扩展系统，还可补充更细致的兼容性和压力测试。

若后续继续优化，可优先从以下方向展开：

1. 补充病害类样本并平衡各类别数据量。
2. 对病害类别进行进一步合并或重定义，降低类间混淆。
3. 增加更多人工测试样例，补充复杂场景下的识别验证。

## 9. 本章小结

本章围绕系统功能、接口、稳定性和实验表现对项目进行了测试与结果分析。测试结果表明，系统页面访问正常、核心接口稳定、识别主流程完整、历史记录和知识库等辅助模块可正常工作。结合训练得到的 `YOLOv11s` 模型结果，当前系统已经具备作为毕业设计最终演示系统的可用性，并能够为论文中的系统测试与实验分析章节提供支撑。
